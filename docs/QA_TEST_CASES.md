# QA Test Cases

이 문서는 코드 단위 테스트 이후, 실제 서비스 화면에서 사용자가 직접 겪는 흐름을 검증하기 위한 수동 QA 기준이다. API 내부 구현 검증이 아니라 브라우저, GitHub OAuth, DB, GitHub API, 화면 상태가 함께 맞물리는 end-to-end 사용자 경험을 확인한다.

## 테스트 범위

- GitHub OAuth 로그인과 로그아웃
- 최초 진입, 인증 필요 페이지, 온보딩 강제 흐름
- 온보딩 설문 작성과 재설정
- 추천 이슈 조회, 필터, 검색, 무한 스크롤, 부분 실패 안내
- 이슈 북마크 추가, 삭제, 목록 반영
- PR 히스토리 조회, 상태 필터, 검색, 통계
- 마이페이지 프로필, 온보딩 요약, 활동 요약
- 인증 만료, GitHub API 실패, DB 실패 등 사용자 관점 오류 상태
- 모바일/데스크톱 반응형과 기본 접근성

## 테스트 전 준비

### 필수 환경

- 프로덕션 또는 staging URL
- GitHub OAuth App의 callback URL 등록
  - 로컬: `http://localhost:3000/api/auth/callback/github`
  - 배포: `https://서비스도메인/api/auth/callback/github`
- 환경변수 설정
  - `DATABASE_URL`
  - `GITHUB_CLIENT_ID`
  - `GITHUB_CLIENT_SECRET`
  - `AUTH_SECRET`
- DB migration 적용
  - `pnpm db:migrate`

### 권장 테스트 계정

- GitHub 계정 A: 공개 저장소와 PR 이력이 있는 계정
- GitHub 계정 B: 공개 저장소/PR 이력이 거의 없는 신규 또는 빈 계정
- GitHub 계정 C: 이미 온보딩을 완료한 기존 계정

### 브라우저/기기

- Chrome 최신 버전
- Safari 또는 Edge 중 1개
- 모바일 viewport
  - iPhone SE 폭
  - 일반 모바일 폭 390px 내외
- 데스크톱 viewport
  - 1280px 이상

## QA 판정 기준

- Pass: 사용자가 다음 행동을 이해할 수 있고, 데이터가 정상 저장/조회되며, 새로고침 후에도 상태가 유지된다.
- Fail: 화면이 깨지거나, 사용자가 진행할 수 없거나, 저장/조회 결과가 실제 상태와 다르다.
- Blocked: 외부 GitHub 장애, 테스트 계정 권한 부족, staging 환경 미구성처럼 테스트 자체를 진행할 수 없다.
- Needs Review: 동작은 가능하지만 메시지, 로딩, 빈 상태, 반응형 등 사용자 경험이 애매하다.
